Backend actual
--------------
Archivos clave:

- server/index.js
  Configura Express, CORS, JSON, ruta /health, monta /api/auth, /api/ventas y /api/reportes. Llama connectAll() antes de escuchar en PORT.

- server/config/db.js
  Mantiene un mapa de conexiones MongoDB: hq, s01, s02, ..., s10. Lee URIs MONGO_HQ y MONGO_S01...MONGO_S10 desde .env. Implementa connectAll(), getConn(key), getConnBySucursal(sucursalId). Esto es clave para demostrar distribucion fisica por cluster.

- server/middleware/auth.js
  Implementa auth JWT, checkBranch y soloGerentes. Cajero no ve reportes. Admin puede acceder a cualquier sucursal. Gerente debe quedar limitado a su sucursal.

- server/routes/auth.js
  Login por username/password/sucursalId. Si sucursalId es 0 busca en HQ; si es 1-10 busca en el cluster de esa sucursal. Genera JWT con userId, username, rol, sucursalId.

- server/routes/ventas.js
  POST /api/ventas: registra venta en cluster de la sucursal autenticada, consulta catalogo en HQ, valida stock en inventario de sucursal, descuenta inventario y crea documento Venta.
  GET /api/ventas/sucursal/:sucursalId: historial con filtros desde/hasta/limite.

- server/routes/reportes.js
  GET /api/reportes/ventas-por-sucursal: consulta en paralelo multiples sucursales y regresa totalVentas, numVentas, ticketPromedio y error por sucursal.
  GET /api/reportes/ventas-por-dia/:sucursalId
  GET /api/reportes/productos-top/:sucursalId
  GET /api/reportes/metodos-pago/:sucursalId

- server/models
  Producto, Venta, Inventario, Sucursal, Usuario. Los schemas cubren lo basico de la rubrica.

- server/seed/seed.js
  Genera 200+ productos, 10 sucursales, admin HQ, gerente y cajeros por sucursal, inventario por sucursal y 200+ ventas por sucursal. Usuarios demo detectados:
  admin / admin1234 en sucursal 0
  gerente_s01 / gerente1234, gerente_s02 / gerente1234, etc.
  cajero1_s01 / cajero1234, cajero2_s01 / cajero1234, etc.

Brechas backend importantes
---------------------------
1. No hay endpoint publico/protegido para listar catalogo de productos desde HQ.
   El POS obliga a teclear SKU manualmente, lo cual complica la demo. Crear GET /api/productos con busqueda por texto, sku, categoria, paginacion y activo=true.

2. No hay endpoint de inventario consultable por sucursal para frontend.
   Crear GET /api/inventario/sucursal/:sucursalId, con filtros por texto, bajoStock=true, categoria si se cruza contra productos HQ, limite y paginacion.

3. No hay endpoint de estado de nodos para demostrar arquitectura distribuida.
   Crear GET /api/nodos/status que indique hq/s01...s10: conectado, nombre, sucursalId, latencia aproximada si se puede medir con ping simple, conteos basicos opcionales. Esto seria muy pro para la demo.

4. En POST /api/ventas la actualizacion de inventario se hace dentro de un ciclo y no es transaccional entre lectura/venta/descuento. MongoDB multi-cluster no permite transaccion global simple; documentar esto. Como mejora minima, validar todos los productos y stocks primero, despues descontar y crear venta. Si falla a mitad, puede haber inconsistencia. Para demo puede bastar, pero conviene reducir riesgo.

5. No hay ruta de anulacion/devolucion de venta. No es obligatoria por rubrica, pero es una mejora profesional si hay tiempo: POST /api/ventas/:id/cancelar que restaure stock y marque venta cancelada.

6. No hay tests automatizados. Agregar pruebas minimas o al menos una coleccion Postman documentada para auth, venta, historial y reportes.

7. server/package.json solo tiene test placeholder. Agregar scripts:
   - dev: node --watch index.js
   - start: node index.js
   - seed: node seed/seed.js

8. Revisar CORS para despliegue. Ahora app.use(cors()) permite todo. Para entrega esta bien en demo, pero documentar variable CLIENT_URL y configuracion segura opcional.

Brechas frontend prioritarias
-----------------------------
Esta es la parte mas importante a mejorar porque el sistema ya tiene backend util, pero la UI no demuestra suficiente para una evaluacion visual y funcional.

1. POS demasiado basico.
   Falta catalogo visible, busqueda por SKU/nombre/codigo de barras, seleccion rapida de productos, mostrar stock disponible, precio unitario, subtotal por item, total antes de pagar, validacion visual de stock insuficiente y resumen tipo ticket.

2. No hay historial de ventas del turno.
   La rubrica/documento pide historial de ventas del turno para cajero activo. Backend ya tiene GET /api/ventas/sucursal/:sucursalId. Implementar panel lateral o seccion inferior con ultimas 10-20 ventas, fecha, cajero, total, metodo, numero de items y boton ver detalle.

3. No hay filtros de fecha en reportes.
   Backend soporta desde/hasta en varios endpoints. UI debe tener filtros Hoy, Semana, Mes, Personalizado y aplicarlos a ventas por sucursal y ventas por dia. Esto ayuda a cumplir reporte diario/semanal/mensual.

4. No hay selector multi-sucursal visual para admin.
   Backend soporta query sucursales=1,2,3. UI podria permitir seleccionar todas, regiones o sucursales especificas. Esto demuestra consultas en paralelo a 5+ clusters.

5. No se muestran errores parciales por cluster.
   /ventas-por-sucursal regresa error por sucursal. La UI debe mostrar chips o tabla de estado: OK/Error por nodo. Esto demuestra tolerancia parcial y arquitectura distribuida.

6. No hay vista explicita de arquitectura distribuida.
   Agregar pagina "Nodos" o seccion en dashboard con HQ + 10 sucursales, estado conectado, colecciones principales y contador/latencia opcional. Aunque sea informativa, se ve muy profesional y ayuda mucho en entrevista.

7. Responsividad incompleta.
   App usa sidebar fija de 220px y grids de 4 columnas; puede romper en pantallas pequenas. Agregar layout responsive: sidebar colapsable o topbar en movil, grids 1/2/4 columnas, charts con alturas adaptadas.

8. Estilos inline excesivos.
   No es fatal, pero dificulta mantenimiento. Se puede migrar gradualmente a componentes reutilizables: Button, Card, Input, Select, Badge, PageHeader, EmptyState, LoadingState, ErrorBanner.

9. Accesibilidad y pulido.
   Falta aria-label en botones icono, labels asociados, estados focus consistentes, contraste revisado, mensajes de error persistentes en reportes, skeletons/loading states, empty states.

10. No hay exportacion visual de graficas.
   El documento pide exportacion visual de graficas en pantalla. Puede interpretarse como mostrar graficas, pero para verse pro agregar boton exportar PNG o CSV en reportes. Minimo exportar CSV de resumen.

11. No hay datos auxiliares de demo visibles.
   Agregar en login un bloque discreto de credenciales demo o boton "usar admin demo", "usar gerente S01", "usar cajero S01" puede acelerar la presentacion. Si se considera inseguro, dejarlo solo en modo desarrollo con import.meta.env.DEV.

12. No hay proteccion fina en frontend por rol.
   App oculta Reportes a cajero, pero POS sigue visible para admin y gerente. Definir comportamiento esperado: admin puede no vender porque sucursalId 0 no tiene cluster de ventas. Si admin entra al POS, POST /ventas puede fallar porque getConnBySucursal(0) no existe. Solucion: si rol admin, default pagina reportes y ocultar/deshabilitar POS o permitir seleccionar sucursal operativa para venta demo.
